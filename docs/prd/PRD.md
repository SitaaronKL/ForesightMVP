# Product Requirements: Foresight Care Operating System (MVP)

**Author:** Dhruv Lalwani
**Prepared for:** Foresight Health interview (Tienlan Sun)
**Date:** May 14, 2026
**Status:** MVP scope for 48-hour build window
**Companion docs:** CCM_Plan.md, APCM_Plan.md, Technical_Architecture.md, UIUX_Spec.md

---

## Goal

Build a deployable Minimum Viable Product that demonstrates how a single registered nurse can supervise a panel of 500 or more patients under Advanced Primary Care Management (or 400 under Chronic Care Management plus Principal Care Management) using a combination of operational design and artificial intelligence augmentation. The Minimum Viable Product is a showcase, not a production system; correctness, polish, and narrative coherence trump comprehensive coverage and edge-case handling.

The Minimum Viable Product must demonstrate three things end-to-end:

1. The panel feels tractable. A nurse can open the application and immediately understand what to do today, in what order, and why. 500 patients are not overwhelming when surfaced correctly.
2. The artificial intelligence is integrated, not bolted on. The agent appears in the workflow at moments where it removes friction, not as a separate chat surface tacked on to the side.
3. The regulatory layer is a first-class object. Audit-ready documentation falls out of the workflow as a byproduct, not as a separate step.

## In-scope flows (6)

### Flow 1: Nurse Dashboard

The primary workspace. The home screen the nurse sees after logging in.

**Layout (top to bottom)**

- **Header bar** with practice name, nurse name, profile menu, search affordance
- **Key Performance Indicator strip**: panel size, patients reached this month, reach rate, average documentation time, service-element coverage percentage
- **Today's queue** (above the fold): the agent-recommended priority order for today. Card-row hybrid. Each row shows patient name, tier (Level 1, 2, or 3 with a colored badge), last touched date, top open issue, why it is prioritized
- **Full panel** (below the fold or accessed via tab): smart sorted list of all panel patients, default sort by urgency, filterable by tier, status, last touched, reach status, condition
- **Right rail**: the Master Agent (Flow 2), persistent across all screens

**Smart sort default order**

1. Patients with care transition deadlines in the next 48 hours
2. Patients overdue from prior month
3. Patients due this week
4. Patients due this month, sorted by tier (Level 3 first)
5. Stable patients on routine cadence

**User stories**

- "As a nurse, when I log in, I see exactly what I should do today without having to think."
- "As a nurse, I can filter my panel to find any subset (e.g., all Level 3 patients overdue for hemoglobin A1c) in under five seconds."
- "As a nurse, I can see at a glance how my month is going against my reach rate goal."
- "As a nurse, I can drill into any patient from any row in one click."

**Acceptance criteria**

- Dashboard loads in under 500ms on a panel of 500 patients
- Key Performance Indicator strip updates in real time when the underlying data changes
- Today's queue is generated by the agent (calls `generateMorningBriefing`) and cached for the day
- All sort columns are clickable to change sort order
- Filters compose (e.g., "Level 2" AND "overdue" AND "diabetes")
- The "Reach rate" Key Performance Indicator color-codes (green above 80%, yellow 70-80%, red below 70%)

**Out of scope (MVP)**

- Saved filter views
- Custom dashboard layout per nurse
- Drag-and-drop reordering of today's queue
- Print or export

### Flow 2: Master Agent Right Rail

The persistent artificial intelligence surface that follows the nurse across the application.

**Layout**

- Always-visible right rail, collapsible to a 48px-wide icon strip
- When expanded, 360px wide, full height of the viewport minus the header
- Liquid glass aesthetic: backdrop blur extra large, subtle gradient mesh background, soft shadow on the left edge
- Header: agent name (default "Sage"), status dot (green when ready, blue pulse when thinking), collapse arrow
- Body: scrolling message thread with the most recent at the bottom
- Each agent response can contain inline action cards (clickable Apply, Send, Edit, Dismiss)
- Context awareness banner: small chip at top showing current scope ("Context: full panel" or "Context: Mrs. Garcia")
- Input footer: text input, microphone button, attach button (future use)

**On nurse login**

- The right rail expands automatically
- Displays the Morning Briefing as the first agent message: priority queue, heads-up items, yesterday's snapshot
- Includes quick-action buttons: "Start with priority queue," "Customize my day," "Show full panel"

**On nurse end-of-shift**

- A second proactive message: the End-of-Day Wrap
- Reach rate today, automatic retry schedules, unsigned documentation, service-element coverage gaps

**During the day**

- Nurse types or speaks queries: "who needs a refill this week," "summarize Mrs. Garcia's last 3 calls," "draft a SOAP note for Mr. Wallace"
- Agent calls up to four tools per turn, then surfaces "Continue" button if more depth is needed
- Tool calls render inline: "Searching panel..." → "Found 8 patients" → final structured response
- Action suggestions render as clickable cards with confirmation required before any write

**Capabilities (read tools)**

- `getPatient(patientId)`: full patient context
- `listPanel(filters)`: filtered panel list
- `getOverdue(nurseId)`: overdue this month
- `getDueToday(nurseId)`: today's queue
- `getPatientHistory(patientId, days)`: recent encounters, labs, notes
- `getCarePlanHistory(patientId)`: all care plan versions
- `getMonthlyKPIs(nurseId, month)`: dashboard metrics
- `getServiceElementCoverage(patientId, month)`: APCM coverage
- `getRecentTranscripts(patientId, n)`: last n call transcripts

**Capabilities (write tools, all require nurse confirmation)**

- `draftSoapNote(transcriptId | encounterId)`: produces draft, status "draft", awaiting nurse sign
- `suggestCarePlanRevision(patientId, rationale)`: drafts new carePlanVersion, status "pending_review"
- `draftPatientMessage(patientId, intent)`: drafts portal/SMS message text
- `scheduleOutreachAttempt(patientId, method, when)`: queues retry

**Capabilities (action tools, longer-running)**

- `transcribeAudio(audioStorageId)`: Whisper API call, returns transcript
- `generateMorningBriefing(nurseId, date)`: assembles priority queue
- `generateEndOfDayWrap(nurseId, date)`: assembles recap
- `computeRiskScore(patientId)`: on-demand scoring
- `detectPatternsAcrossEncounters(nurseId, timeWindow)`: cross-patient cluster detection
- `scanForGapClosure(panelFilter)`: missing preventive services

**Hard constraints (encoded in system prompt)**

- Agent NEVER speaks to a patient directly in version one
- Agent NEVER auto-submits a claim, message, or care plan update
- Agent is scoped to the calling nurse's panel only (enforced by query layer)
- Audit row created for every suggestion and for every applied action

**User stories**

- "As a nurse, when I log in, the agent tells me what matters today before I have to ask."
- "As a nurse, I can ask the agent anything in natural language and it gives me real answers in seconds."
- "As a nurse, every suggestion the agent makes is a draft I can edit or reject; nothing applies without my click."
- "As a nurse, I can see exactly what the agent is doing under the hood (tool calls visible)."

**Acceptance criteria**

- Right rail visible on every screen
- Morning Briefing renders within 2 seconds of login
- Natural language queries get a response within 5 seconds for cached panel data
- All write tools produce an inline action card with Apply button
- Tool call visualization shows: tool name, arguments, result count, then final agent response
- Voice input via microphone button works in modern browsers
- Tool budget cap is enforced (4 per turn, then "Continue" button)

**Out of scope (MVP)**

- Multi-turn agent reasoning that persists across days (each session is fresh)
- Agent-initiated patient outreach (future via Twilio)
- Custom system prompts per nurse
- Tool calling for non-Foresight data sources (electronic medical record, lab system)

### Flow 3: Patient Detail with Care Plan Version History

The deep view of a single patient. Reachable by clicking any patient row in the dashboard.

**Layout**

- Header: patient name, age, primary nurse, tier, status badge, billing program (CCM / PCM / APCM)
- Tab strip: Overview, Care Plan, Encounters, Documents, Service Elements, Messages
- Right rail: Master Agent, context updated to "Context: [patient name]"

**Overview tab**

- Demographics summary
- Current conditions list with chronic flag
- Active medications
- Recent vitals and labs (last 6)
- Upcoming appointments
- Hospital events in last 90 days
- Risk score with trend line (last 30 days)

**Care Plan tab**

- Renders the CURRENT carePlanVersion
- Sections: problem list, expected outcomes, treatment goals, symptom management, planned interventions, medication management, community/social services, provider coordination, review schedule
- Each section is a structured editable block
- "History" button → opens diff view modal

**Care Plan History (the git-diff view)**

- Modal overlay with timeline of all carePlanVersions on the left
- Selected version on the right, with diff highlighting against the previous version (additions in green, deletions in red, modifications in yellow)
- Each version shows: who drafted it, when, the rationale, the review status, who approved it (if approved)
- Action buttons: "Restore this version" (creates a new version cloning the old), "View full version"

**Encounters tab**

- Chronological list of all encounters for this patient
- Each row: date, type, duration, nurse, status, top topic tag
- Click any encounter to see the full record: SOAP note, transcript (if voice), time log, service-element impact

**Documents tab**

- Medical documents uploaded for this patient (labs, imaging, discharge summaries)
- Click any document to view (PDF or image preview)
- Documents grouped by type
- Version history for documents that have versions (typically not labs; relevant for care plan attachments, advance directives)

**Service Elements tab**

- For Advanced Primary Care Management patients: the 11-element coverage matrix for the current month
- Each element row: name, status (not yet, available, delivered), evidence link
- Filter by month to see prior-month coverage
- "Generate audit packet for this patient" button

**Messages tab**

- Thread of portal messages between patient (or caregiver) and nurses
- Send a new message (with agent-draft option)

**User stories**

- "As a nurse, I can see every piece of context about this patient in under one minute."
- "As a nurse, I can see the full version history of the care plan and exactly what changed in each revision."
- "As a nurse, I can confirm the patient's service-element coverage for the current month at a glance."
- "As a compliance officer, I can pull an audit packet for any patient with one click."

**Acceptance criteria**

- Patient detail loads with all tabs populated in under 800ms
- Care plan diff view highlights added/removed/modified text in distinct colors
- Version timeline supports up to 50 versions without performance degradation
- Service-element matrix updates in real time when an encounter or document is added that satisfies an element
- Audit packet generation produces a PDF with care plan, consent, time logs, encounters, and service-element attestation

**Out of scope (MVP)**

- Editing past care plan versions (always create a new version)
- Document version sub-versioning (each lab is its own document)
- Custom audit packet templates

### Flow 4: Voice-to-SOAP Draft

The documentation drag answer. End-to-end voice capture, transcription, and structured note drafting.

**Trigger**

- Patient detail page, microphone button at the top
- Click to start recording, click to stop
- Visual indicator: pulsing red dot while recording, elapsed time counter

**Recording**

- Browser MediaRecorder API captures audio (webm or mp4 depending on browser)
- Stored locally during recording; uploaded to Convex File Storage on stop

**Processing**

- Upload triggers a Convex Action: `transcribeAudio(audioStorageId)`
- Action calls OpenAI Whisper API
- Transcript returned to the client, appended to the right rail as an agent message
- Second action call: `draftSoapNote(transcriptId)`, returns structured note + time log entry + care plan delta suggestions

**Review modal**

- Modal opens with:
  - Transcript (collapsed by default, expandable)
  - Drafted SOAP note in four editable sections
  - Drafted time log entry (duration, activity description, billable boolean)
  - List of suggested care plan deltas (each with diff preview)
- Each item has Approve, Edit, Reject buttons
- Modal footer: "Sign and save" button (signs the note, writes time log, applies approved deltas)

**User stories**

- "As a nurse, I press one button before a call and one button after; the agent does the rest."
- "As a nurse, the drafted SOAP note is 90% right; I edit 10% and sign."
- "As a nurse, the time log entry is generated automatically with a defensible activity description."
- "As a nurse, the care plan deltas suggested are accurate and easy to accept or reject individually."

**Acceptance criteria**

- Recording starts within 200ms of microphone click
- Transcription completes within 1 second per 10 seconds of audio (Whisper baseline)
- SOAP draft renders within 8 seconds of transcript completion (GPT-4 latency)
- Modal allows editing all four SOAP sections inline
- "Sign and save" creates one encounter row, one timeLog row, one soapNote row with status "signed", and one or more carePlanVersion rows for approved deltas, all in a single transaction
- Audio file persists in Convex File Storage with patient association

**Out of scope (MVP)**

- Multi-speaker diarization (separating nurse from patient voices)
- Live transcription during the call (transcribe after stop)
- Direct Twilio integration for outbound calling (browser microphone only)
- Multiple language support (English only)

### Flow 5: Patient-Facing Portal

The patient's window into their own care.

**Login**

- Lightweight authentication: phone number or email plus one-time code (no passwords)
- Mobile-responsive (a 65+ user base often accesses on mobile)

**Layout**

- Header: practice name, patient name, log out
- Three main sections accessible via tab or large card buttons: My Care Plan, Monthly Statements, Messages

**My Care Plan**

- Plain-English version of the current care plan (not the structured clinical document)
- Sections: My health goals, My medications, My next appointment, Things to watch for, When to call us
- Read-only
- Last updated by [nurse name] on [date]

**Monthly Statements**

- List of monthly statements (one per billed month)
- Click any statement to view detail
- Statement detail: month, nurse name, summary of what was done (3-5 bullet points), Medicare paid, supplemental covered, patient share, balance
- All amounts in dollars and cents
- Plain English throughout

**Messages**

- Thread with the nurse (and optionally physician)
- Patient can send a new message
- Messages appear in the nurse's dashboard as a notification

**User stories**

- "As a patient, I can see exactly what my nurse is doing for me each month."
- "As a patient, I never get a surprise bill because the cost is shown clearly upfront and monthly."
- "As a patient, I can message my nurse asynchronously without having to call."

**Acceptance criteria**

- One-time code authentication works on mobile and desktop
- Care plan summary renders correctly for plans with all sections populated
- Monthly statements pull correct data from billingRecords table
- Messages thread updates in real time when nurse responds
- All language is at or below 8th-grade reading level

**Out of scope (MVP)**

- Patient-side care plan editing or suggestions
- Patient-side appointment scheduling
- Family/caregiver portal access
- Patient-side document upload

### Flow 6: Admin Page

Internal tooling. Not patient-facing, not nurse-facing in normal operation. Lives behind a separate admin role.

**Purpose**

- Manage fake data for the showcase
- Trigger system-level operations manually for the demo

**Layout**

- Sidebar nav: Patients, Nurses, Encounters, Care Plans, Seed Data, System

**Patients tab**

- List of all patients with edit and delete
- Create new patient form
- Bulk seed: "Generate 50 patients" button

**Nurses tab**

- List of all nurses (typically just one for the demo)
- Edit nurse details

**Encounters tab**

- List of all encounters
- Create encounter form with: patient, nurse, type, duration, optional jot notes
- Used during demo to "complete a call" without actually recording one

**Care Plans tab**

- List of all care plans with their version count
- Manually trigger a new version for demo purposes

**Seed Data tab**

- "Generate 50 patients" with realistic demographic distributions
- "Generate 30 days of encounter history" populating the dashboard
- "Reset all data" button (admin only, very destructive, requires confirmation)

**System tab**

- "Recompute all risk scores" (for showcasing the live update during interview demo)
- "Regenerate today's morning briefing" (for re-triggering after schema changes)
- "Run gap closure scan" (for showing the agent surfacing missing services)
- "Trigger end-of-day wrap" (for the bookend demo)

**User stories**

- "As Dhruv (the builder), I can populate the system with realistic mock data in under 30 seconds."
- "As Dhruv during the interview, I can trigger a risk score recomputation live so Tienlan can see the score change."
- "As Dhruv during the interview, I can manually run the gap closure scan and have the agent surface findings."

**Acceptance criteria**

- All seed operations complete in under 10 seconds
- Risk score recomputation is observable from the dashboard (badge color changes in real time via Convex reactivity)
- No admin-only data is accessible to nurse or patient roles

**Out of scope (MVP)**

- Real authentication for admin role (a flag in the user record is sufficient)
- Audit log of admin actions
- Multi-tenant practice management

## Cross-cutting requirements

### Performance

- Dashboard render: < 500ms cold, < 200ms warm
- Patient detail: < 800ms cold, < 300ms warm
- Master agent query response: < 5 seconds for panel queries, < 8 seconds for SOAP drafts
- Voice transcription: < 1 second per 10 seconds of audio
- All mutations: < 200ms (Convex serverless functions)

### Visual polish

- Liquid glass aesthetic: backdrop-blur values, gradient meshes, subtle shadows
- Gradient color palette: primary navy #0B3B5C, accent gradient navy-to-teal, light surface near-white #F8FBFD
- framer-motion for all transitions: 200-300ms ease-out for entries, 150ms for exits
- Subtle micro-interactions: hover scale 1.02, click scale 0.98, focus ring with brand color glow
- Detailed visual spec lives in UIUX_Spec.md

### Accessibility

- Keyboard navigation everywhere
- ARIA labels on all interactive elements
- Color contrast meets WCAG AA on all text
- Focus indicators visible
- Patient portal has larger default font (65+ user base)

### Data accuracy

- All times in patient-facing surfaces shown in the patient's local timezone
- All currencies in dollars and cents with two decimal places
- All dates in patient-facing surfaces use long form ("April 15, 2026" not "4/15/26")

### Showcase-specific behaviors

- One designated demo patient ("Maria Rodriguez") with rich seeded history that demonstrates every flow
- One designated demo nurse ("Sarah Chen") that owns the dashboard view
- Risk score for Maria recomputes on every encounter write for instant demo visibility
- All other patients have nightly risk recomputation only

## Explicitly out of scope (drawn in Excalidraw, not built)

- Twilio outbound calling integration
- Live agentic patient outreach
- Electronic medical record / Fast Healthcare Interoperability Resources integration with Epic, eClinicalWorks, Athena
- Multi-tenant practice management
- Billing clearinghouse submission
- Hospital event feed (Admission, Discharge, Transfer) ingestion
- Quality measure reporting to Centers for Medicare and Medicaid Services
- Mobile native applications (web responsive is sufficient)
- Internationalization or multi-language support
- Comprehensive role-based access control beyond nurse/patient/admin
- Full audit log of all data access
- Patient family/caregiver portal access
- Telehealth video integration

## Success criteria for the showcase

- Tienlan watches the morning briefing render and immediately understands the operational thesis
- Tienlan can ask the agent any question about the panel and get a coherent answer
- Tienlan watches a voice-to-SOAP draft happen end to end and sees the documentation drag answer
- Tienlan clicks into a patient and sees the care plan version history; understands the audit-trail story
- Tienlan flips between the CCM patient view and the APCM patient view and sees the regulatory difference visualized
- Tienlan logs into the patient portal and sees the consent-time clarity / monthly statement approach
- The deployment is live, accessible via URL, and stable for the duration of the interview
- Total flows demonstrated: 6
- Time to demonstrate everything: under 8 minutes (Tienlan will not sit through more)
